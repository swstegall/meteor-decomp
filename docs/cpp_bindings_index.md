# Engine C++-bound Lua API — corpus inventory

> Last updated: 2026-05-03 — this is the index doc; the canonical
> auto-generated reference is at `build/wire/cpp_bindings.md`
> (regenerated by `make extract-cpp-bindings`).

## What this captures

Per `docs/world_master_decomp.md`, the engine declares its
C++-implemented Lua-callable methods using the
`_<method>_cpp` / `_<method>_inl` pair convention in `_u`-suffix
files (encoded as `_p` in the ciphered tree). The `_inl` body is a
2-line stub returning the literal strings `"self"` and
`"_<method>_cpp"`; the engine reads those literals at call time and
dispatches to the matching C++ implementation (per the
`LuaActorImpl` 90-slot binding from Phase 6 item #6).

`tools/extract_cpp_bindings.py` walks every `*_p.luac` file under
`build/lpb/` and scans each file's string table for `_<name>_inl`
patterns. Each match is one C++-bound method on the parent class.

## How to regenerate

```bash
make decode-lpb              # populates build/lpb/*.luac (one-time)
make extract-cpp-bindings    # → build/wire/cpp_bindings.{json,md}
```

`make lpb-corpus` runs all three steps (decode + decompile +
extract) in sequence.

## What this enables for garlemald

This inventory describes the **CLIENT-side engine API surface** —
the methods exposed to the binary's Lua VM for the shipped client
scripts (`*.lpb` files under `client/script/`).

### Important correction (2026-05-03 via `docs/player_base_decomp.md`)

An earlier version of this doc claimed garlemald's `userdata.rs`
needed to converge toward this API surface (suggesting a "~120 method
gap" for PlayerBase). **That framing was wrong.** Garlemald's
`userdata.rs` is the binding for garlemald's OWN server-side mlua
VM — it serves garlemald's own Lua scripts, NOT the client's.

The two API surfaces serve independent VMs:

- **Engine PlayerBaseClass / WorldMaster / etc.** (this inventory) →
  consumed by **client-side** shipped `.lpb` scripts in the binary's
  Lua VM
- **Garlemald LuaPlayer / LuaActor / LuaWorldManager / etc.**
  (in `userdata.rs`) → consumed by **server-side** garlemald scripts
  in mlua

The actionable garlemald binding work is documented in
`docs/garlemald_lua_coverage_index.md` — that's the report
cross-referencing what garlemald's scripts call against what
garlemald's userdata.rs binds. The 87 gaps there are the real
coverage gaps, NOT the 419 engine bindings here.

### What this inventory IS useful for, garlemald-side

1. **Knowing what state fields the client expects to read.** Every
   engine method here implicitly references some `playerWork.<field>`
   or `charaWork.<field>` that garlemald MUST populate via
   SetActorProperty packets. The method names hint at what fields
   must exist (e.g. `_getAchievementRate` → `playerWork.achievement*`
   field family).
2. **Knowing what server-RPC pairs the client expects.** Methods like
   `_callServerOnTalk` are the client side of garlemald's `onTalk`
   server dispatch. Each `_callServer*_cpp` method here corresponds
   to a wire opcode garlemald should expect to receive.
3. **Knowing what's PURELY client-side.** Methods like
   `_setKeyboardFocusedWidget`, `_saveUserConfig`, `_lockCameraControl`
   never touch the server — garlemald shouldn't try to drive these.

## Most actionable findings

### Top "missing" PlayerBase methods garlemald likely needs

(Sample from the full list; complete reference at
`build/wire/cpp_bindings.md`.)

- **Talk lifecycle**: `_executeTalk`, `_canExecuteTalk`,
  `_cancelTalk`, `_breakTalk`
- **Emote lifecycle**: `_executeEmote`, `_canExecuteEmote`,
  `_cancelEmote`, `_breakEmote`
- **Notice lifecycle**: `_cancelNotice`, `_breakNotice`
- **Push lifecycle**: `_cancelPush`, `_breakPush`
- **Server callbacks**: `_callServerOnTalk`, `_callServerOnEmote`,
  `_callServerOnPush`, `_callServerOnCommand`
- **Event mode**: `_isEventPlaying`
- **GM**: `_getGMRank`

These mirror the `onTalk`/`onPush`/`onEmote` server-side dispatch
garlemald already drives — the C++-bound methods are the
client-side terminations of those flows.

### desktopWidget surface = the UI control plane

The 43 desktopWidget methods cover **widget container ops, target
cursor control, user config persistence, user macro CRUD, and
target-tutorial flow control**. This is the canonical UI-control
API. Garlemald-side, the relevant packets are:
- Widget-create/destroy → mapped to `_createWidgetInWidgetContainer`
  / `_deleteCreatingWidgetInWidgetContainer`
- Target update → `_setTargetCharacter` /
  `_setTargetCharacterByDisplayName`
- User config / macro persistence → garlemald serves these from
  the per-player save state

### `math` re-shimming is intentional

Lua's standard `math` library (32 functions: `_abs`, `_acos`,
`_asin`, `_atan`, `_atan2`, `_byte`, etc.) is **re-shimmed as
engine bindings** rather than just exposed via Lua's stdlib. This
is likely so the engine can:
1. Deterministically replay math ops (for client/server consistency)
2. Substitute fixed-point or platform-stable implementations where
   Lua's float-based defaults would diverge across hardware

Garlemald's mlua-based binding gets Lua's stdlib `math` for free —
no action needed unless quest scripts depend on FFXIV-specific
math behaviour we haven't observed yet.
